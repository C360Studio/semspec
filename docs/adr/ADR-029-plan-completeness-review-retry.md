# ADR-029: Plan Completeness Review and Retry Loop

**Status:** Accepted
**Date:** 2026-03-29
**Deciders:** Coby, Claude

## Context

The plan-reviewer validates plans against SOPs (compliance) but does not check structural **completeness** — whether requirements cover the goal, scenarios test the requirements, and the dependency graph is valid. Plans that are SOP-compliant but incomplete pass review and reach humans or execution with gaps.

Additionally, when review returns `needs_changes`, the plan transitions to `StatusRejected` — a dead end with no retry mechanism. The `ReviewIteration` and `MaxReviewIterations` fields exist on the Plan struct but are not wired into any automation.

## Decision

### 1. Add completeness validation to plan review

The plan-reviewer runs two review rounds. Each round validates SOP compliance (existing) **and** structural completeness (new):

**Round 1 (plan document — after drafting):**
- Goal clarity — specific and actionable, not vague
- Context sufficiency — enough context for requirements to be derived
- Scope validity — files exist and are relevant to the goal

**Round 2 (requirements + scenarios — after generation):**
- Goal coverage — requirements collectively address the stated goal
- Requirement→Scenario coverage — every requirement has at least one scenario
- Dependency validity — `depends_on` references exist, form a valid DAG (no cycles, no orphan refs)
- No orphaned scenarios — every scenario references an existing requirement ID
- Scope alignment — scope files are relevant to the requirements

Completeness failures are `severity: "error"` findings that block approval, same as SOP violations.

### 2. Implement a retry loop for rejected plans

The review + retry loop is entirely **pre-human**. The reviewer is a quality gate ensuring only complete plans reach the human approval step. Once a human has approved at either gate, the automated reviewer does not re-trigger for that stage.

#### Pipeline with human gates and retry zones

```
created → planner → drafted
  → [REVIEWER R1 ↔ retry zone] → reviewed → [HUMAN APPROVES] → approved
  → req-gen → scenario-gen → scenarios_generated
  → [REVIEWER R2 ↔ retry zone] → scenarios_reviewed → [HUMAN APPROVES] → ready_for_execution
```

#### Retry behavior per round

**Round 1 (plan document):**
- On `needs_changes`: send `plan.mutation.revision` (round=1)
- Mutation handler increments `ReviewIteration`, stores findings, resets status to `StatusCreated`
- **Keeps existing Goal/Context/Scope intact** — the planner refines the draft rather than starting from scratch
- Planner prompt detects populated Goal + ReviewFindings → switches to refinement mode ("revise this draft based on feedback")

**Round 2 (requirements + scenarios):**
- On `needs_changes`: send `plan.mutation.revision` (round=2)
- Mutation handler increments `ReviewIteration`, stores findings, resets status to `StatusApproved`
- **Clears Requirements/Scenarios** — they will be re-generated by requirement-generator and scenario-generator
- Human approval of the plan document (Round 1 gate) stands — R2 only retries the decomposition
- Generator prompts include review findings so the same gaps are not repeated

**Iteration cap:** `MaxReviewIterations` (default 3). When exhausted, the plan transitions to `StatusRejected` (truly terminal — human must intervene).

### 3. New mutation: `plan.mutation.revision`

A single mutation that handles both rounds:

```go
type RevisionMutationRequest struct {
    Slug     string          `json:"slug"`
    Round    int             `json:"round"`     // 1 or 2
    Verdict  string          `json:"verdict"`   // "needs_changes"
    Summary  string          `json:"summary"`
    Findings json.RawMessage `json:"findings"`
}
```

Handler logic:
1. Increment `ReviewIteration`
2. Store `ReviewFindings`, `ReviewSummary`, `ReviewFormattedFindings`
3. If under iteration limit: set status to re-entry point (Round 1 → `StatusCreated`, Round 2 → `StatusApproved`)
4. If at limit: set `StatusRejected` with `LastError`

### 4. State transition additions

| From | To | Purpose |
|------|----|---------|
| `StatusRejected` | `StatusCreated` | R1 retry — planner refines draft |
| `StatusRejected` | `StatusApproved` | R2 retry — re-generate reqs/scenarios (already exists) |

## Consequences

**Positive:**
- Plans reaching humans are structurally complete, not just SOP-compliant
- Automated retry reduces human intervention for fixable issues
- Planner refinement mode preserves work instead of discarding drafts
- Existing `ReviewIteration` / `MaxReviewIterations` fields are now functional
- Iteration cap prevents infinite loops

**Negative:**
- Additional LLM calls per plan (up to 2 extra review rounds × 2 stages)
- Retry loop adds latency before human sees the plan
- Completeness checks rely on LLM judgment — may produce false positives

**Risks:**
- Planner refinement may not converge — same feedback repeated across iterations. Mitigated by iteration cap.
- Round 2 retry clears requirements/scenarios — if only scenarios were incomplete, requirement re-generation is wasted work. Acceptable tradeoff for simplicity; future optimization could target only the incomplete stage.
